# Plan: react-to-lit — Standalone React-to-Lit Component Transpiler

**Vision:** A CLI tool that takes React component source and produces Lit web components. Cloudscape is the first target — 100+ components generated, not hand-written.

## Architecture

```
@cumulus-ui/react-to-lit (standalone package — new repo)
  ├── src/
  │   ├── parser/        — TypeScript AST: read React components
  │   ├── transforms/    — React pattern → Lit pattern (the core)
  │   ├── emitter/       — Produce Lit TypeScript source
  │   └── cli.ts         — CLI entry point
  └── test/              — Transform-level unit tests

@cumulus-ui/components (consumer — existing repo)
  ├── src/               — GENERATED by react-to-lit
  ├── scripts/           — Thin wrapper calling react-to-lit
  └── overrides/         — Hand-written behavior patches (if any)
```

## The Core Transform Map

Every React pattern has a Lit equivalent. The transpiler handles each:

### Component Structure
- `function Component(props: Props)` → `class CsComponent extends CsBaseElement`
- `export default Component` → `@customElement('cs-component') export class ...`
- `forwardRef((props, ref) => ...)` → class with `@query` for ref forwarding

### Props & State
- Props interface → `@property()` declarations with correct types/attributes
- `useState(initial)` → `@state() private _name = initial`
- `useReducer(reducer, init)` → `@state()` + reducer method
- Destructured props → class properties

### Lifecycle
- `useEffect(() => { ... }, [])` → `connectedCallback()` (mount-only)
- `useEffect(() => { ... }, [deps])` → `willUpdate(changed)` with dep checks
- `useEffect(() => { return cleanup })` → `disconnectedCallback()` for cleanup
- `useLayoutEffect(...)` → `updated()`
- `useMemo(() => val, [deps])` → getter with caching or `willUpdate` pre-computation

### Refs & DOM
- `useRef<HTMLElement>(null)` → `@query('selector') private _el!: HTMLElement`
- `ref.current` → `this._el`
- `useRef(value)` → private field (non-DOM refs)

### Events
- `onClick={handler}` → `@click=${this._handler}`
- `onKeyDown={handler}` → `@keydown=${this._handler}`
- `onChange` callback prop → `fireNonCancelableEvent(this, 'change', detail)`
- `onFollow` callback prop → `fireNonCancelableEvent(this, 'follow', detail)`

### JSX → html Templates
- `<div className={clsx(styles.root, active && styles.active)}>` → `html\`<div class=${classMap({root: true, active: this.active})}>\``
- `{condition && <Child />}` → `${condition ? html\`<cs-child>\` : nothing}`
- `{items.map(i => <Item key={i.id} />)}` → `${items.map(i => html\`<cs-item>\`)}`
- `<>{children}</>` → `<slot></slot>`
- `<div>{header}</div>` → `<div><slot name="header"></slot></div>` (for SlotContent props)
- `dangerouslySetInnerHTML` → `unsafeHTML()`

### CSS
- `import styles from './styles.css.js'` → already generated by `generate-styles.ts`
- `styles.root` → string class name from the CSS module
- `clsx(styles.a, cond && styles.b)` → `classMap()` directive

### Custom Hooks → Reactive Controllers
- `useControllable(value, handler, default)` → `ControllableController` (already built)
- `useMergeRefs(ref1, ref2)` → manual ref forwarding
- `useUniqueId()` → static counter or `crypto.randomUUID()`
- `useVisualRefresh()` → removed (always visual refresh in Lit)
- Custom hooks with state → `ReactiveController` subclass

### Form Integration
- Components with `name`, `value`, `disabled` → extend `FormControlMixin`
- `useFormField()` hook → `FormControlMixin` provides this
- `setCustomValidity()` → `ElementInternals.setValidity()`

## Phases

### Phase 1: Parser — Read React components into an IR

Build an intermediate representation (IR) that captures the component structure without React-specific details.

- [ ] Define the IR types: `ComponentIR`, `PropIR`, `StateIR`, `EffectIR`, `HandlerIR`, `TemplateNodeIR`
- [ ] Parse a React function component into `ComponentIR` using TypeScript AST
- [ ] Extract props from the interface (reuse `component-meta.ts` logic)
- [ ] Extract `useState` calls → `StateIR`
- [ ] Extract `useEffect` calls → `EffectIR` with dependency analysis
- [ ] Extract `useRef` calls → `RefIR`
- [ ] Extract event handler functions → `HandlerIR`
- [ ] Parse JSX return → `TemplateNodeIR` tree
- [ ] Test: parse Cloudscape's `Badge`, `Spinner`, `StatusIndicator` (simple components)

### Phase 2: Emitter — Produce Lit TypeScript from IR

- [ ] Emit class declaration with correct base class and decorator
- [ ] Emit `@property()` declarations from `PropIR`
- [ ] Emit `@state()` declarations from `StateIR`
- [ ] Emit lifecycle methods from `EffectIR`
- [ ] Emit event handlers from `HandlerIR`
- [ ] Emit `render()` method from `TemplateNodeIR` → Lit `html` template
- [ ] Emit style imports
- [ ] Test: generate Badge, Spinner, StatusIndicator — must compile and render

### Phase 3: Transforms — Handle complex patterns

- [ ] `clsx()` / `classnames()` → `classMap()`
- [ ] Conditional rendering → ternary with `nothing`
- [ ] List rendering → `.map()` in template
- [ ] Slot detection (SlotContent props) → `<slot>` elements
- [ ] Event callback props → `fireNonCancelableEvent`
- [ ] `useControllable` → `ControllableController`
- [ ] `forwardRef` → ref forwarding pattern
- [ ] Test: generate Checkbox, Input, Button — must compile and render

### Phase 4: Deduplication — Extract shared patterns

- [ ] Identify repeated code across generated components (shared hooks, utilities)
- [ ] Extract into shared controllers, mixins, utilities
- [ ] `FormControlMixin` — already exists, wire generated components to use it
- [ ] `ControllableController` — already exists, wire generated components to use it
- [ ] Common keyboard handlers → shared utility
- [ ] Test: generate 20 components, verify shared code is deduplicated

### Phase 5: Full generation — All 100+ components

- [ ] Generate all Cloudscape components
- [ ] Run build — all must compile
- [ ] Run existing tests — all must pass
- [ ] Diff against hand-written versions — identify behavioral gaps
- [ ] Create override mechanism for components that need hand-written patches

### Phase 6: Override mechanism

For the ~20% of behavior that needs human touch:

- [ ] Define override pattern: `src/{component}/overrides.ts` that patches the generated class
- [ ] Generator preserves overrides on re-generation
- [ ] Overrides are typed — they extend the generated class
- [ ] Test: override a generated component, re-generate, verify override survives

## What This Replaces

Once react-to-lit generates all components:

| Current | After |
|---|---|
| `generate-interfaces.ts` | Subsumed — react-to-lit produces interfaces |
| `generate-styles.ts` | Kept — CSS extraction is separate concern |
| `generate-render.ts` | Subsumed — react-to-lit produces render templates |
| `generate-permutations.ts` | Kept — permutations are test infrastructure |
| `component-meta.ts` | Reused — metadata layer feeds the transpiler |
| Hand-written `internal.ts` | REPLACED — generated from React source |

## Success Criteria

- `npx react-to-lit --input vendor/cloudscape-source/src/badge --output src/badge` produces a working component
- All 100+ Cloudscape components compile
- Existing tests pass without modification
- Re-running the generator produces identical output (deterministic)
- Override mechanism allows hand-written patches that survive re-generation

## Out of Scope (for v1)

- Non-Cloudscape React libraries (future — the tool is general-purpose but v1 targets Cloudscape)
- Server-side rendering
- React context → Lit context (use prop drilling for v1)
- Concurrent mode / Suspense patterns
- CSS-in-JS (Cloudscape uses CSS modules, which we already handle)
